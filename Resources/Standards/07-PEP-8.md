# PEP 8 — Style Guide for Python Code

The Python language's official style guide. Authored by Guido (the language's creator), updated as the language evolves. Almost universally followed in Python ecosystems.

## Major sections

### Code Layout
- **4 spaces per indent level.**
- **Maximum line length 79 chars** (or 99 if your team agrees; PEP 8 explicitly allows extension).
- Continuation: hanging indent or align with opening delimiter.
- Two blank lines between top-level functions/classes; one blank line between methods.

### Imports
- One per line.
- Order: standard library → related third-party → local.
- Absolute imports preferred; relative imports allowed for intra-package.
- No `from x import *` (except in `__init__.py` for explicit re-export).

### String Quotes
- Single or double quotes — pick one and be consistent.
- Triple double-quotes for docstrings (PEP 257).

### Whitespace
- Avoid extraneous whitespace inside parens/brackets/braces.
- Single space around binary operators (with exceptions for arithmetic precedence).
- No spaces around `=` in keyword arguments (except when annotated).

### Comments
- Complete sentences, capitalized, period at end.
- Block comments aligned with the code they describe; `# `.
- Inline comments separated by ≥2 spaces, with `# `.
- Docstrings for all public modules, functions, classes, methods (PEP 257).

### Naming Conventions
- **`lower_case_with_underscores`** for functions, variables.
- **`CapWords`** (PascalCase) for classes.
- **`UPPER_CASE_WITH_UNDERSCORES`** for constants.
- **`_single_leading_underscore`** for "weakly private" (convention).
- **`__double_leading_underscore`** for name-mangled private.
- **`__double_leading_and_trailing_underscore__`** for "magic"/dunder methods (don't invent your own).

### Programming Recommendations
- `is None` / `is not None`, not `== None`.
- `if not seq` for empty sequences (idiom).
- Use exception chaining (`raise X from Y`).
- Specific exceptions over bare `except:`.
- Use `with` for resource management.
- String methods over the `string` module.
- Function annotations (PEP 484) for type hints.

### Type Hints (PEP 484)
- Annotate public APIs.
- Use `Optional[X]` for `X | None` (or `X | None` directly in 3.10+).
- Forward references with strings or `from __future__ import annotations`.

## Tooling
- **Black** — opinionated formatter, slightly more aggressive than PEP 8 (88-col default).
- **autopep8 / yapf** — PEP 8 formatters with more configurability.
- **flake8 / pycodestyle** — linters.
- **Ruff** — modern fast linter implementing PEP 8 + many other lint rules.
- **mypy / pyright** — type-hint checkers.

## Where PEP 8 is silent
- Architectural choices.
- Library/framework conventions (Django, NumPy, etc.).
- Most teams adopt PEP 8 + Black + a project-specific addendum.

## Pairs with
- **PEP 257** — docstring conventions.
- **PEP 484** — type hints.
- **PEP 20** — The Zen of Python (philosophical, not stylistic).
